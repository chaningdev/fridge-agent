# 設計判断ログ（Design Decisions）

ポートフォリオ向けに、「なぜそう作ったか」を記録するドキュメント。
実装の What（何を作ったか）は README に、Why（なぜそうしたか）をここに書く。

---

## アーキテクチャ

### マルチベンダー構成（Gemini + OpenAI）を採用した理由

**判断:** 画像認識は Gemini、判断・対話は OpenAI GPT と役割を分けた。

**理由:**
- Gemini 2.5 Flash は画像の細かい文字認識（レシート）が得意で、料金も安い
- GPT は自然文の意図理解と Tool 選択（function-calling）の精度が高い
- 単一モデルに集約すると得意不得意が出る。タスク特性でモデルを使い分けるのが、LINEヤフー「Agent i」と同じ設計思想

**トレードオフ:** APIキーが2つ必要になる。OpenAI 残高切れ時のフォールバックを `router.py` のルールベース判定で担保した。

---

### フォールバック設計（LLM なし → ルールベース）

**判断:** `OPENAI_API_KEY` が未設定 or 残高切れの場合、GPT を呼ばずキーワード判定で動作させる。

**理由:**
- デモや開発中に API 残高を消費しすぎない
- オフライン環境でも在庫の確認・追加・消費は動く
- エラーで止まるより、機能を絞ってでも動き続ける方がユーザー体験が良い

---

### router.py をツールと agent の間に挟んだ理由

**判断:** `agent.py` が直接 `tools.py` を呼ぶのではなく、`router.py` を経由させた。

**理由:**
- どのツールがどのバックエンド（gemini/openai/sqlite）を使うかを1箇所（`TOOL_BACKENDS`）で管理できる
- Streamlit の UI 上で「このツールは Gemini が処理しました」と表示するトレース情報を一元管理できる
- 将来モデルを差し替えるときに router.py だけ変えればよい

---

## データベース

### SQLite を選んだ理由

**判断:** PostgreSQL や Firebase ではなく SQLite を使った。

**理由:**
- 個人・家族スケールの在庫管理に外部 DB サーバーは不要
- ファイル1つ（fridge.db）で完結するのでセットアップが簡単
- 将来スケールが必要になったとき、`db.py` の接続部分だけ差し替えれば移行できる設計にした

---

### history テーブルを inventory と分けた理由

**判断:** 在庫の現在値（inventory）と操作履歴（history）をテーブルごと分けた。

**理由:**
- 在庫を上書きしてしまうと「いつ何を使ったか」が分からなくなる
- 履歴があることで「先週何をよく使ったか」「食材をどこで買ったか」の分析ができる
- イベントソーシングに近い設計で、在庫の不整合をデバッグしやすい

---

## ツール設計

### 各 Tool を「単体でテストできる設計」にした理由

**判断:** `tools.py` の各関数は、agent.py や router.py に依存せず単独で呼べる。

**理由:**
- Gemini や OpenAI をモックすれば API 不要でユニットテストが書ける
- 1つの Tool が壊れても他の Tool は動き続ける
- 将来 FastAPI や LINE Bot のバックエンドとして使いまわせる

---

### レシートの商品名を汎用名に正規化した理由

**判断:** Gemini に返させる食材名を「産地・ブランドを除いた汎用名」に統一した。

**理由:**
- レシートには「淡路玉ねぎ」「四国玉ねぎ」のように産地付きで記載されることが多い
- 商品名そのままで在庫登録すると、同じ食材が別エントリになって在庫が分散する
- 在庫管理の目的は「何が何個あるか」であり産地情報は不要
- プロンプトに Few-shot 例を含めることで Gemini の正規化精度を安定させた

**トレードオフ:** 産地・鮮度情報が失われる。必要であれば `notes` フィールドを追加して保持できる。
